# 翻译规则样例参考

本文件只保存具体正反样例和实操提醒。主 Skill 负责流程、边界和抽象方法；本文件用于在对应步骤中渐进式读取。所有样例都使用抽象占位符，不代表任何游戏、插件、指令编码、文件名、路径层级或标签名的固定规则。

## 使用边界

- 进入术语候选、占位符收束、插件规则、事件指令规则或 Note 标签规则步骤时，只读取当前步骤的小节。
- 样例只说明判断方法，不是当前游戏答案。最终判断必须回到当前工作区 JSON、CLI 校验结果和用户明确提供的信息。
- 不要把样例字段名、路径层级、指令编码、标签名、插件参数名或文本类型当作通用规律。
- 如果当前游戏样本和样例不同，以当前样本为准；不确定时排除或报告风险，不靠样例补脑。

## 术语候选样例

术语候选必须保持输入类别和 key 不变，只填写 value。候选文件只是主代理审查输入，不能直接代替最终字段译名表或正文术语表。

```text
输入片段: {"item_names": {"【<组织名>は<对象>を一掃している】": "", "<药品原名>": ""}}
正确候选: {"item_names": {"【<组织名>は<对象>を一掃している】": "【<组织译名>正在清剿<对象译名>】", "<药品原名>": "<自然中文药品名>"}}
错误候选: {"item_names": {"【<组织名>は<对象>を一掃している】": "【<组织音译><对象音译>希特伊鲁】"}, "note": "已处理"}
```

判断重点：

- 正确候选保留了类别和 key，只填写 value。
- 错误候选新增了说明字段，并用机械音译替代语义翻译。
- 字段包装符号是否保留，必须看当前游戏字段用途；不要因为样例保留或重组了符号，就套到所有术语上。

## 占位符规则样例

占位符规则用于保护必须原样保留的游戏协议片段。规则必须只遮蔽协议本体，不能吞掉玩家可见文本。

```text
输入候选: {"marker": "\\X[<资源标识>]", "covered": false}
正确规则: {"(?i)\\\\X\\[[^\\]\\r\\n]+\\]": "[CUSTOM_PLUGIN_X_MARK_{index}]"}
错误规则: {"CUSTOM_PLUGIN_X": "(?i)\\\\X\\[[^\\]\\r\\n]+\\]"}
错误规则: {"(?i)\\\\N\\d*": "[CUSTOM_PLUGIN_N_{index}]"}
```

混合协议文本样例：

```text
原文片段: \FXStop this!!!
已确认协议: \FX
正确方向: {"\\\\FX": "[CUSTOM_PLUGIN_FX_MARKER_{index}]"}
预览要求: 模型可见文本仍保留 Stop this!!!
错误方向: 保护 \FXStop，导致 Stop 被吞掉。
```

判断重点：

- JSON 键是正则表达式，值是 `[CUSTOM_...]` 模板；不要写反。
- 裸控制符紧贴正文时，先确认协议边界，再只保护协议本体。
- 宽规则如果会匹配普通转义、换行或可翻译正文，必须收窄。
- `validate-placeholder-rules --sample <原文片段> --input <规则文件> --json` 的预览里，去掉 `[CUSTOM_...]` 后仍应看到需要翻译的玩家可见文本。

## 三类外部规则样例

三类外部规则只决定哪些外部字符串叶子进入正文翻译集合。占位符规则会在这些文本进入集合后保护协议片段，但不能替代外部规则筛选。

### 插件规则

```text
输入片段: {"name": "<插件名>", "parameters": {"message": "按钮文本", "entries": [{"label": "菜单项"}], "file": "<资源目录>/<资源文件名>.png", "count": "12"}}
正确输出: [{"plugin_index": 0, "plugin_name": "<插件名>", "paths": ["$['parameters']['message']", "$['parameters']['entries'][*]['label']"]}]
错误输出: {"<插件名>": ["$['parameters']['message']"]}
错误输出: [{"plugin_index": 0, "plugin_name": "<插件名>", "paths": ["$.parameters.message", "$['parameters']['file']", "$['parameters']['count']"]}]
```

判断重点：

- 只选玩家可见文本。
- JSONPath 使用括号语法并从 `$['parameters']` 开始。
- 资源路径、脚本、数字、颜色、布尔值和内部标识都排除。
- 插件参数名是否代表可见文本，必须看当前 `plugins.json` 的值和上下文。

### 事件指令规则

```text
嵌套对象输入片段: {"code": <指令编码>, "parameters": ["<插件名>", "<命令名>", "显示名文本", {"messageText": "提示文本", "file": "<资源文件名>.png"}]}
正确输出: {"<指令编码>": [{"match": {"0": "<插件名>", "1": "<命令名>"}, "paths": ["$['parameters'][3]['messageText']"]}]}
顶层字符串输出: {"<指令编码>": [{"match": {"0": "<插件名>", "1": "<命令名>"}, "paths": ["$['parameters'][2]"]}]}
单字符串输入片段: {"code": <另一个指令编码>, "parameters": ["<命令名> text:提示文本 file:<资源文件名>.png"]}
单字符串输出: {"<另一个指令编码>": [{"match": {}, "paths": ["$['parameters'][0]"]}]}
错误输出: {"<指令编码>": [{"match": {"plugin": "<插件名>"}, "paths": ["$['parameters']['messageText']"]}]}
```

判断重点：

- 事件指令编码必须以当前 `event-commands.json` 顶层编码为准。
- 参数结构可能是嵌套对象、顶层字符串或单个命令字符串，必须按当前导出样本判断真实 JSONPath。
- 不要照搬某个引擎、某个插件或某个样例的位置。
- `match` 的键必须是参数索引字符串。
- 如果单字符串同时含协议壳和玩家可见文本，完成报告必须提醒主代理后续用占位符规则保护协议片段。

### Note 标签规则

```text
输入片段: {"file_name": "<物品数据文件>.json", "tag_name": "<玩家可见说明标签>", "sample_values": ["药品的详细说明文本"]}
输入片段: {"file_name": "<地图文件模式>", "tag_name": "<玩家可见名牌标签>", "sample_values": ["向导"]}
输入片段: {"file_name": "<装备数据文件>.json", "tag_name": "<机器协议标签>", "sample_values": ["1,2,3"]}
输入片段: {"file_name": "<技能数据文件>.json", "tag_name": "<玩家可见补充说明标签>", "sample_values": ["技能追加说明文本"]}
正确输出: {"<物品数据文件>.json": ["<玩家可见说明标签>"], "<技能数据文件>.json": ["<玩家可见补充说明标签>"], "<地图文件模式>": ["<玩家可见名牌标签>"]}
错误输出: {"<物品数据文件>.json": ["<机器协议标签>"], "<装备数据文件>.json": ["<内部状态编号标签>"]}
错误输出: {"note": {"<物品数据文件>.json": ["<玩家可见说明标签>"]}}
```

判断重点：

- 长段自然语言说明、地图事件名牌、任务提示、状态说明等通常可能是玩家可见文本。
- 数字编号、脚本公式、资源路径、开关枚举、装备状态编号、连锁技能编号等通常是机器协议或系统标签。
- 输出只能是 `{data文件名或文件模式: [note标签名, ...]}`，不写标签正则、不写 reason、不改游戏 `data/*.json`。
- 同一个标签名在不同游戏里含义可能不同，必须以当前候选的 `sample_values` 为准。
