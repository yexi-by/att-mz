# 外部文本规则流程

外部文本规则负责把插件参数、事件指令参数和 Note 标签中的玩家可见文本纳入正文翻译集合。规则只决定哪些字符串叶子进入翻译；进入后仍要由占位符规则保护其中的协议片段。

本阶段采用审查型工作流。插件规则、事件指令规则和 Note 标签规则分别由工作子代理主动发现和整理，再由审查子代理检查误选、漏选、空结果证据、过拟合和路径真实性。主代理写 `review-decisions/external_rules.json` 后，才允许 validate/import。

## 执行顺序

1. 术语表导入完成后，启动第二轮外部规则分析。
2. 插件规则、事件指令规则和 Note 标签规则可以并行处理，三个工作任务不能写同一个最终规则文件。
3. 工作子代理可在 `<工作区>/agent-scratch/external_rules/<任务ID>/` 写一次性脚本，统计字符串叶子、字段形态、JSON 字符串内部叶子、空结果证据和混合协议文本样本。
4. 工作子代理完成报告写入 `<工作区>/agent-reports/external_rules/<任务ID>.json`，必须包含交叉验证摘要，说明选中项依据、重点排除项依据、空结果依据、主动发现和仍需主代理确认的风险。
5. 派发对应审查任务，或在低风险项目中派发聚合 `external_rules_review`。审查报告写入 `<工作区>/review-reports/external_rules/<任务ID>.json`。
6. 主代理等待全部工作和审查结果，逐项检查结构、可见文本判断、交叉验证摘要、空结果理由和 findings。
7. 主代理抽样核对每类规则的选中项、重点排除项和空结果；证据不足或存在未关闭 `blocker` 时修规则或要求重做，不直接导入。
8. 主代理写 `<工作区>/review-decisions/external_rules.json`。
9. 裁决为 `approved` 后分别运行对应 `validate-*`。
10. 校验通过后运行对应 `import-*`。空结果必须用对应空结构并在导入时确认。
11. 三类外部规则全部导入后，才能重新生成和收束占位符规则。

非标准 data 文本和插件源码文本默认只在高风险且用户确认后处理；低风险项目只有在用户明确要求时才启动。非标准 data 使用 `nonstandard-data-risk-report.json`、`nonstandard-data/candidates.json` 和非标准 data 任务契约；插件源码使用 `plugin-source-risk-report.json` 和插件源码任务契约。这两类都不并入三个默认子代理任务。

派发子代理时不能只概括“分析规则”。必须复制对应任务契约，填入 `<发行版目录>`、`<工作区>`、`<游戏标题>`、输入文件、唯一可写文件、审计材料目录和校验命令。需要具体正反样例时，只读取 `translation-rule-examples.md` 中对应小节，并提醒子代理：样例只说明判断方式，不是当前游戏固定答案。

外部规则审查必须检查：误选资源、脚本、机器协议或内部标识；因为字段复杂而漏选玩家可见文本；JSONPath 是否来自真实候选；空结果是否有证据；混合协议文本是否提醒后续占位符保护；是否过拟合某个插件、编码或样例；插件配置是否变化。误选会破坏运行协议、路径不存在、空结果无证据或候选范围不完整时必须给 `blocker`。

## 插件规则

- 输入：`plugins.json`、`plugin-json-string-leaf-candidates.json`。
- 输出：`plugin-rules.json`，格式为 `[{plugin_index, plugin_name, paths}]`。
- 只选择玩家会在界面、菜单、对话、提示、任务、状态或说明中看到的自然语言文本。
- 排除资源路径、文件名、脚本、公式、数字、颜色、坐标、布尔值、枚举值、开关名、内部标识、调试字段和纯配置键。
- 外层 JSON 字符串字段不能直接当路径规则，必须判断解析后的内部字符串叶子。
- 必须交叉验证：用 `plugins.json` 的字段名、相邻字段、参数值形态和 `plugin-json-string-leaf-candidates.json` 互相核对；必要时只读对应 `js/plugins/<插件名>.js` 直接源码文件中的插件头注释、`@param`、`@text` 和 `@desc`，确认字段语义。
- 插件源码只读交叉验证不得扫描 `js` 根目录、不得递归子目录、不得修改 JS、不得直接写规则以外文件。
- 合法空结果是 `[]`。
- 子代理任务契约见 `plugin-rules-agent-task.md`。

## 事件指令规则

- 输入：`event-commands.json`。
- 输出：`event-command-rules.json`，格式为 `{指令编码字符串: [{match, paths}]}`。
- 工作区可能包含默认导出的插件命令编码，也可能包含 CLI 显式指定的任意事件指令编码；必须以当前导出的顶层编码为准。
- 空规则导入必须传 `--confirm-empty`；如果工作区使用了显式 `--code`，导入空规则时必须传同一组 `--code CODE`。
- 只选择玩家可见自然语言文本；排除资源、脚本、数字、布尔值和内部标识。
- 单个字符串叶子可能混合命令语法和玩家可见文本。可见文本明确且协议边界稳定时，可以纳入规则，并在报告中提醒主代理后续用占位符规则保护协议片段。
- 必须交叉验证：用同一编码下的参数形态、重复出现次数、相邻参数、对象键名和值分布互相核对；无法从导出文件确认玩家可见性时排除并报告风险。
- 合法空结果是 `{}`。
- 子代理任务契约见 `event-command-rules-agent-task.md`。

## Note 标签规则

- 输入：`note-tag-candidates.json` 和 `note-tag-rules.json` 草稿。
- 输出：`note-tag-rules.json`，格式为 `{data文件名或fnmatch文件通配模式: [note标签名, ...]}`。文件键可用 `Map*.json` 通配，不是正则。
- 只选择 `note` 字段里由插件消费且玩家可见的文本，例如物品说明、技能说明、任务说明、地图事件名牌或提示语。
- 必须根据 `sample_values` 判断，不能把某个标签名当成所有游戏通用答案。
- 必须交叉验证：用标签名、`sample_values`、data 文件分布、同标签多样本和已回填草稿互相核对；样例不足以确认玩家可见性时排除并报告风险。
- 排除脚本、公式、资源名、ID、布尔/枚举、数值列表、资源引用、内部关联字段、开关名、坐标和文件路径。
- 合法空结果是 `{}`。
- 子代理任务契约见 `note-tag-rules-agent-task.md`。

## 插件文本触发键

第二轮分析时，主代理必须主动检查插件或脚本是否把名字框、地图显示名、数据库名称、系统类型或其他文本型字符串当作功能触发键使用。

项目层不阻止地图显示名、数据库名称、系统类型和标准名字框汉化。如果当前游戏插件依赖原始字符串触发语音、立绘、状态列表或其他功能，应在游戏目录内新增或修改插件侧映射，让“中文显示值 -> 原始触发值”的关系留在游戏侧。游戏侧映射只服务当前游戏，不写进 A.T.T MZ 项目核心。
