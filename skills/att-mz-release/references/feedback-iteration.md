# 试玩反馈迭代

试玩反馈是正式翻译流程的一部分。静态质量检查不能替代实际游玩。收到反馈后，先定位真实文本和根因，再补规则、补译文、重新检查并按需再次写进游戏文件。不要为了修一个反馈无证据扩大规则，也不要把所有问题都当成“加规则”处理。

## 收集信息

优先向用户收集：

- 问题截图或原文片段。
- 游戏场景。
- 玩家看到的现译文。
- 期望表达。
- 是否影响继续游玩。
- 能否稳定复现。

如果用户只说有些地方没翻或翻译很怪，先请用户补充截图、具体文本或场景；不能凭空猜测并批量重置。

## 分类处理

- 漏翻：先运行 `translation-status`、`quality-report` 和必要时的 `audit-active-runtime`，确认是没成功保存译文、未纳入规则的文本来源、写入缺口，还是当前运行文件残留。
- 多翻/过翻：如果资源名、脚本、触发键、内部标识、插件协议、Note 标签协议或本应保留的源语言被翻译，先定位是外部规则误选、插件源码 selector 误选、MV 名字框误伤、占位符规则过窄，还是结构化占位符边界错误；不要只改译文或继续扩大规则。
- 明显错译、称呼错、语气不自然、术语不统一：优先导出修复表或手动填写译文表，只改中文译文行后导入；如果是术语体系缺口，再补术语并复查受影响译文。
- 控制符、名牌、语音触发标记、自动替换标记异常：先按占位符或结构化占位符规则风险处理，必要时精确重置受影响文本。
- 菜单、插件界面、事件指令参数或 Note 标签仍是源语言文本：重新准备或复用工作区，检查对应外部规则。只有候选来源、玩家可见证据和排除项审查都成立时才补规则。
- 游戏窗口放不下、换行难看、按钮被截断：按质量检查报告和用户截图修短中文译文行，再导入并复查。
- 图片文字、视频文字或引擎外资源文字：记录为当前自动文本流程外的问题，除非用户另开资源修图任务。

## 补规则边界

- 规则只能解决“文本来源纳入/排除”或“协议片段保护”问题，不能用来掩盖错译、漏译、整句源文残留或模型质量问题。
- 漏翻不能无脑加宽插件、事件指令、Note 标签、非标准 data 或插件源码规则；必须先证明该字符串是玩家可见文本，并列出同类排除项。
- 多翻/过翻优先收窄或删除误选规则，并精确重置受影响译文；不要用源文保留例外或占位符规则掩盖整段误纳入。
- 单个反馈如果暴露的是一类文本来源缺口，先补对应外部规则；如果只是某条译文不好，修译文，不改规则。
- 外部规则、MV 虚拟名字框规则、非标准 data 规则或插件源码规则改变后，必须重新执行占位符收束；占位符规则改变后，必须重新检查受影响译文。

## 规则级联

补充或收窄插件规则、事件指令规则、Note 标签规则、非标准 data 规则、插件源码规则或 MV 虚拟名字框规则后，按顺序执行：

1. 运行对应 `validate-*` 和 `import-*`，不能绕过 CLI。
2. 重新运行 `build-placeholder-rules`、`validate-placeholder-rules`、`scan-placeholder-candidates` 和 `import-placeholder-rules`。
3. 如存在固定协议外壳包住显示文本，重新运行结构化占位符规则校验、扫描和导入。
4. 运行 `rebuild-text-index --game <游戏标题>`，让当前文本索引和 current text facts 反映新规则。
5. 对新增 pending 或误纳入后需要重翻的文本，运行 `translate` 或用 `reset-translations --input <文件>` 精确重置。
6. 运行 `audit-coverage` 和 `quality-report`，确认没有规则缺口、占位符风险、失效译文或质量 error。

## 反馈反查

用户提供原文清单后，运行：

```powershell
.\att-mz.exe verify-feedback-text --game <游戏标题> --input <反馈原文清单>
```

反查仍存在的文本按分类处理：

- 规则缺口：补插件规则、事件指令规则、Note 标签规则、非标准 data 规则或插件源码规则；补完必须执行“规则级联”。
- 译文缺口：继续翻译或导出手动填写译文表。
- 写入缺口：重新检查覆盖审计和写回流程。
- 插件源码硬编码：用 `scan-plugin-source-text` 生成翻译源风险报告；怀疑写回后的当前运行文件有漏翻、坏控制符或 JS 语法错误时，用 `audit-active-runtime` 直接审计当前文件。需要处理文本时，按开局支线策略处理；默认自动按插件源码支线导出 AST 地图，只有用户已选择跳过或处理成本/风险超出策略时才询问。

## 闭环要求

每轮反馈修复都走：

```text
反馈清单 -> 定位根因 -> 补译文或补/收窄规则 -> 规则级联（如规则变化） -> audit-coverage -> quality-report -> 用户确认是否再次写进游戏文件 -> write-back 或 rebuild-active-runtime -> audit-active-runtime -> verify-feedback-text
```

如果反馈影响范围很小，优先精确修复对应文本。如果反馈显示某类规则整体漏掉，先补规则再处理相关文本，并按规则级联重建索引和占位符保护。只有用户明确选择完整重译时，才执行完整重译。

每轮再次写进游戏文件后，说明这一轮反馈已处理，请用户继续试玩确认。直到用户确认主要流程已玩过、关键问题已处理，或明确接受剩余问题。

用户确认本轮试玩反馈已经处理完成，或明确表示暂不继续修复后，才清理 `<工作区>`；如果工作区由 `prepare-agent-workspace` 生成，优先用 `cleanup-agent-workspace --workspace <工作区>`。
