# 性能与重型计算底线专项 Review 计划

本文档只定义性能与重型计算底线专项 review 的计划和验收边界，不是执行方案，也不包含任何已执行结论。

## 1. 目标范围

专项审查 `att-mz` 中所有会遍历或处理以下对象的流程：

- 当前游戏 `data/*.json`
- 当前游戏 `js/`，包括插件配置、插件源码、脚本片段
- 当前游戏数据库中的术语、规则、已保存译文记录
- 文本范围、候选扫描结果、质量检查结果、写回计划
- 所有会影响吞吐、并发、内存、超时、取消、进度、失败恢复的路径

最终输出三类内容：

- 问题清单与优先级路线图
- 测试驱动性能报告
- 架构审计报告

## 2. 硬前置顺序

1. 只读契约梳理

   只允许读取 CLI 帮助、README、已有测试、公开命令说明，用来确认伪造样本必须覆盖哪些 RPG Maker MZ 结构。此阶段禁止开始性能问题审查，禁止下性能结论。

2. 伪造重型游戏样本

   必须先生成日文伪游戏和英文伪游戏。两者都必须是伪造数据，不使用真实游戏内容。生成完成前，禁止开始 review。

3. 启动专项 review

   样本存在后，才允许进入全流程动态巡检、源码审计、profiling、问题归档。

## 3. 伪造样本要求

样本类型：

- 日文伪游戏
- 英文伪游戏

规模：

- 每个伪游戏目标 `500MB-1GB`
- 只包含项目参与处理的内容：
  - `data/*.json`
  - `js/`
- 不包含图片、音频、字体、视频、存档、无关资源

持久化策略：

- 超重样本本地生成，不提交
- 保留生成器或生成说明作为可复现资产
- 不保留临时计数器、profiling 辅助、伪 OpenAI 服务等 review 工具
- 生成产物应被 `.gitignore` 或等效机制排除

结构覆盖：

- 大量 `MapXXX.json`
- 大量事件、事件页、事件指令
- `101/401/102/405/357` 等文本相关事件命令
- 长文本、短文本、重复原文、跨行文本、超长行
- 日文假名、片假名、汉字、全角符号、控制符混合
- 英文短词、缩写、专名、长段英文、标点密集文本
- `Actors/Classes/Skills/Items/Weapons/Armors/Enemies/States/System/CommonEvents/Troops` 等数据库文本
- 大量 Note 标签
- 大量插件配置 `$plugins`
- 多个大型插件源码 `.js`
- JSONPath 规则可命中的深层字符串叶子
- 自定义控制符、脚本标记、占位符、疑似候选、确认风险场景
- 故意包含少量协议易损文本，用于质量检查压力测试

## 4. 本地伪模型服务

review 计划必须包含本地 OpenAI 兼容伪服务：

- 不消耗真实模型额度
- 覆盖 `translate` 和 `run-all`
- 日文伪游戏返回中文伪译文
- 英文伪游戏返回中文伪译文
- 可控地产生：
  - 大多数合格译文
  - 少量坏译文
  - 占位符损坏
  - 残留源文
  - 超宽行
  - 输出格式错误
- 用于验证保存、质量检查、失败恢复、进度和重试表现

## 5. 动态 Review 模式

采用全流程巡检型 review。

每个相关 CLI 命令都要记录：

- 是否加载 `GameData`
- 加载次数
- 文本范围构建次数
- 候选扫描次数
- 插件配置扫描次数
- 插件源码或 AST 解析次数
- JSONPath 规则匹配次数
- 数据库译文记录扫描次数
- 写回计划构建次数
- 是否复用同一轮中间结果
- 是否重复全量扫描
- CPU 密集任务是否在 Python 串行执行
- Rust 原生扩展是否承接重活
- 并发参数是否真实参与调度
- `ATT_MZ_RUST_THREADS` 或配置入口是否生效
- 命令耗时、峰值内存、CPU 利用率
- 超时、取消、进度、日志、失败恢复表现

## 6. 命令覆盖范围

按“所有可能遍历游戏内容的 CLI 命令”全覆盖，包括但不限于：

- 注册或准备游戏工作区
- 文本扫描与诊断
- 术语导出、校验、导入
- 占位符规则校验、导入
- 插件配置导出
- 插件规则校验、导入
- 事件指令参数导出
- 事件指令规则校验、导入
- Note 标签或类似规则流程
- `translate`
- `write-back`
- `run-all`
- 质量报告
- 覆盖报告
- 写回前检查
- 写回后审计
- 重置译文
- 数据库清理或恢复相关命令

## 7. 证据采集方式

采用内建计数与外部 profiling 组合：

- review 期间允许临时 instrumentation 或测试钩子
- 统计重复加载、重复扫描、重复构建、重复解析
- 使用外部 profiling 记录耗时、峰值内存、CPU 利用率
- 所有临时工具只服务 review，不保留
- 最终报告记录证据来源和复现步骤

## 8. 缺陷分级

### P0

- 极限伪游戏下主流程无法完成，且不是输入非法导致
- 写回或质量检查在大规模输入下出现数据损坏、错误放行或错误阻断
- 并发、取消或失败恢复问题导致不可控资源占用或不可恢复状态

### P1

- 同一命令内无必要地重复全量加载或扫描当前游戏文本
- 重复构建文本范围、候选集合、写回计划或大索引
- CLI 并发参数存在但没有真实参与调度
- 大规模候选扫描、AST/JSONPath、质量检查、hash/索引构建仍由 Python 串行承担，且无规模边界说明
- `translate`、`write-back`、`run-all` 不能复用同一轮已构建中间结果
- Rust 并行线程数不可控或 `ATT_MZ_RUST_THREADS` 不生效

### P2

- 局部重复扫描但影响范围有限
- 有性能风险但仅在极限样本下明显
- 缺少足够日志、进度、profiling 入口或性能说明
- 暂留 Python 实现但已明确规模边界和迁移触发条件不足

## 9. 报告结构

最终 review 报告建议分为：

1. 样本说明

   日文和英文伪游戏生成方式、规模、目录结构、覆盖点。

2. 巡检矩阵

   每个命令一行，列出耗时、内存、扫描次数、复用情况、并发情况、是否命中缺陷。

3. 问题清单

   按 P0/P1/P2 排序。每条包含：发生了什么、影响什么、证据、根因判断、建议修复方向、验证方式。

4. 架构审计

   检查 `GameData`、文本范围、候选事实、数据库译文记录、写回计划是否存在单一事实来源和跨阶段复用边界。

5. 路线图

   先修 P0/P1，再处理 P2。每项说明是否需要 Rust 迁移、并发调度、缓存复用、契约调整或测试补强。

6. 剩余风险

   明确哪些命令未能完整跑、哪些证据只来自静态审计、哪些 profiling 结果受本机环境影响。

## 10. 禁止事项

- 伪造样本完成前禁止开始性能 review
- 禁止使用真实游戏内容
- 禁止消耗真实模型额度
- 禁止为了通过检查而缩小样本到不具备压力价值
- 禁止只凭源码感觉定性性能问题
- 禁止把临时计数器、profiling 辅助、伪服务作为业务实现长期保留
- 禁止提交超重伪游戏产物
- 禁止把无关资源放进伪游戏样本
- 禁止把“确认风险”误当成允许协议片段被翻坏

## 11. 后续执行入口

后续执行这份 review 时，第一步必须是只读契约梳理，然后设计并生成日文和英文极限伪游戏。样本生成完成前，不进入任何性能问题审查。
