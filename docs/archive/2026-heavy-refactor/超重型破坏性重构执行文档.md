## 0. 执行前内置上下文

本章是本文的自包含背景。后续执行者可直接从本文开始阶段任务；如果本文和源码现状冲突，以当前源码、测试和公开外部契约为准，并把冲突写入阶段验收记录区。

### 0.1 项目定位

A.T.T MZ 是面向 RPG Maker MV/MZ 游戏汉化流程的 Python CLI 工具。它围绕“注册干净游戏、建立可信源快照、导出规则工作区、导入规则、调用模型翻译、检查译文、写入当前运行文件、根据试玩反馈补漏”这一链路工作。

核心业务链路：

```text
干净游戏目录
-> 注册并创建可信源快照
-> 导出工作区候选和规则草稿
-> 导入并审查文本规则、占位符规则、术语规则
-> 构建当前可翻译文本范围
-> 调用模型翻译并保存已验证译文记录
-> 运行质量检查和写文件前检查
-> 从可信源快照、规则和已保存译文记录生成当前运行游戏文件
-> 根据试玩反馈继续补漏
```

### 0.2 当前系统边界

| 边界 | 当前含义 | 重构时必须保持或重建的判断 |
| --- | --- | --- |
| 可信源快照 | 注册游戏时创建的 `data_origin`、`js/plugins_origin.js`、`js/plugins_source_origin` | 正文翻译和普通写回以可信源快照为来源，当前运行文件不能被当作源文事实 |
| 当前运行文件 | 游戏实际运行的 `data`、`js/plugins.js`、`js/plugins` | 它是写入目标和运行状态审计对象，损坏时应显式失败、重建或诊断 |
| SQLite 游戏库 | 每个游戏一套数据库，保存元数据、规则、术语、译文、文本索引、运行记录和写回映射 | schema 版本必须精确匹配；旧库不静默兼容 |
| 工作区 JSON | Agent/用户处理候选、规则、手动译文和修复表的临时文件 | 工作区是交换载体，不是内部事实来源；导入成功后才成为数据库事实 |
| LLM 输出 | 模型返回的批量译文 JSON | 模型输出不可信，保存前必须经过 ID、行数、结构、占位符和源文残留等校验 |
| Rust 原生能力 | CPU 密集扫描、JS AST、写回协议、质量检查和写回计划 | Python 负责编排、I/O、路径安全和报告；Rust 负责重型计算；跨语言契约必须有统一版本门槛 |
| Skill 契约 | 开发版和发行版 Agent 翻译流程 | Skill 只描述翻译流程；普通源码开发、重构和测试修复不使用翻译流程 Skill |
| 发行包 | GitHub Actions `release` 工作流构建的 Windows ZIP | 本机不能作为正式发行构建和验收环境 |

### 0.5 重构判断准则

后续阶段遇到局部修复诱惑时，按以下问题判断是否必须提升为结构性重构项：

1. 主流程是否表达当前正确模型，而不是历史问题合集。
2. 新能力是否真正替代旧路径；旧路径是否仍进入默认流程。
3. 重要业务结论是否只有一个事实来源。
4. 质量门禁、安全门禁、写入门禁是否重复且口径漂移。
5. 文件职责是否过密，导致多个业务概念挤在同一处。
6. 公共抽象是否缺失，或公共函数已经变成万能工具。
7. 文件树是否表达当前系统结构，而不是历史堆积。
8. 测试是在保护业务行为，还是锁死坏结构。
9. 文档、Skill、README 和发行脚本是否描述当前实现。
10. 性能优化是否带来减法：减少重复扫描、重复状态和重复判断。

判断结论：

- 如果局部补丁会保留双事实来源、隐藏状态同步、旧兼容成功路径或重复门禁，不做局部补丁，必须升为结构性重构。
- 如果旧数据库、旧规则、旧工作区或旧 native 扩展不符合当前契约，默认显式失败，不自动转换、不静默兼容。
- 如果一个参数、配置或环境变量存在，就必须完成“定义 -> 解析 -> 校验 -> 应用 -> 测试”全链路。
- 如果一个命令会遍历当前游戏全部文本、插件配置、插件源码、AST、数据库译文或写回计划，必须记录本命令内全量扫描次数，并优先复用中间结果。

## 1. 重构总目标

这次破坏性重构要解决的不是某几个函数难看，而是当前系统的正确模型被历史兼容、重复事实来源、重复门禁、重复扫描和膨胀测试遮住。重构完成后，系统必须重新表达一个清楚、唯一、可验证的当前模型：

```text
干净游戏目录
-> 注册并创建可信源快照
-> 基于本轮 manifest 导出工作区
-> 导入规则并在同一事务中更新规则、审查状态和受影响译文
-> 构建唯一当前 TextScope
-> 调用模型翻译并保存通过校验的译文
-> 基于同一质量结果生成报告和写文件前检查
-> 基于可信源快照、规则和已保存译文生成写回计划
-> 在可回滚边界内写入当前运行文件和诊断映射
-> 根据试玩反馈继续补漏
```

重构总目标分为六个可验收结果：

1. 当前模型唯一：每个核心业务结论只有一个权威产出位置，报告、门禁、测试只能消费它，不再各自推导。
2. 历史路径退出默认流程：旧 hash、旧环境变量、未注册写回补快照、旧工作区残留、质量错误快路径等不再作为默认成功路径。
3. 失败语义一致：命令失败时，数据库、当前运行文件、字体副作用、运行映射和用户 JSON 报告描述同一个状态。
4. 性能路径可解释：全量扫描、AST 解析、质量检查和写回计划在单个命令内有明确复用边界；重复扫描必须能被测试或基线发现。
5. Python/Rust 边界稳定：Python 负责配置、I/O、事务、报告和编排；Rust 负责 CPU 密集扫描、AST、质量和写回计划；跨语言契约有统一版本硬门槛。
6. 测试保护业务行为：测试不再主要锁死历史入口和大文件 helper，而是保护公开 CLI/Agent 主链路、外部 JSON、事务失败、性能预算和发行包边界。

非目标：

- 不追求单纯减少行数；减少行数只是删除旧路径和重复模型后的结果。
- 不做本地发行包验收；正式发行仍只能由 GitHub Actions `release` 工作流构建。
- 不保留隐式兼容作为“温和过渡”；不兼容时显式失败，并给出中文下一步。

## 2. 当前系统问题总览

本文内置的 22 个确认问题不是孤立 bug，而是下列结构问题的具体证据。

| 结构问题 | 对应问题 | 当前表现 | 重构判断 |
| --- | --- | --- | --- |
| 写入和数据库事务边界分裂 | ATT-MZ-REVIEW-010、ATT-MZ-REVIEW-015、ATT-MZ-REVIEW-016 | 字体覆盖、写后审计、规则导入跨表更新都可能在命令失败后留下局部副作用 | 不能局部补 `try/except`；必须重建写回和规则导入的事务模型 |
| Agent JSON、质量报告和门禁事实来源重复 | ATT-MZ-REVIEW-003、ATT-MZ-REVIEW-014 | 正常报告用 `AgentReport`，顶层错误手写 envelope；写回级质量错误和细分计数互相矛盾 | 必须让报告消费同一个结构化结果，不保留并行摘要 |
| 工作区、规则、TextScope 的当前事实被旧状态分裂 | ATT-MZ-REVIEW-007、ATT-MZ-REVIEW-011、ATT-MZ-REVIEW-013、ATT-MZ-REVIEW-022 | legacy hash、旧工作区文件、质量错误历史记录、配置覆盖索引都能参与当前判断 | 必须建立本轮 manifest、当前 RuleSet、当前 TextScope 的单一事实链 |
| 高风险支线门禁依赖调用方携带临时事实 | ATT-MZ-REVIEW-009 | 插件源码扫描在 prepare 场景阻断，但直接 translate/rebuild-text-index 可能跳过 | Workflow gate 必须自己拥有高风险扫描和阻断事实 |
| 旧源文模型仍能进入测试和兼容路径 | ATT-MZ-REVIEW-006 | 缺可信源快照时从当前运行文件补建源快照 | 必须删除未注册写回补快照模型；注册才是源文事实入口 |
| 运行限制参数没有真实参与调度 | ATT-MZ-REVIEW-012 | `--stop-on-error-rate` 达阈值后仍等待已排队模型任务结束 | 必须重建翻译取消和队列生命周期，不做消费者局部抛错 |
| Python/Rust 契约不统一 | ATT-MZ-REVIEW-017 | JS AST adapter 不校验 native contract version | 所有 Rust 热路径必须走同一个 native contract 硬门槛 |
| 跨域协议放在错误模块名下 | ATT-MZ-REVIEW-008 | 事件指令 JSONPath 依赖 `app/plugin_text/paths.py` | 必须重建中性 JSON 参数路径协议边界 |

## 3. 目标系统模型

重构后的系统用下列对象表达当前模型。每个对象都有唯一权威生产者，其他模块只能消费或展示。

| 当前模型对象 | 权威生产者 | 允许消费者 | 不允许的旧行为 |
| --- | --- | --- | --- |
| `SourceSnapshot` 可信源快照 | `add-game` 注册流程和 manifest 校验 | 文本抽取、TextScope、写回计划、恢复当前运行文件 | 写回或测试 helper 从当前运行文件补建源快照 |
| `WorkspaceManifest` 本轮工作区清单 | `prepare-agent-workspace` | `validate-agent-workspace`、`cleanup-agent-workspace`、规则导入命令 | 按固定文件是否存在启用旧可选支线 |
| `RuleSet` 当前规则集 | 规则导入事务 | TextScope、workflow gate、写回计划、Skill/CLI 报告 | legacy hash、部分候选 hash、导入失败后半更新 |
| `TextScopeSnapshot` 当前文本范围 | TextScope 服务或 text index 重建结果 | translate、manual import、quality-report、write-back、coverage | quality_error 历史记录直接当当前范围 |
| `WorkflowGateResult` 前置门禁结果 | `collect_workflow_gate_errors` 的重建版本 | translate、run-all、manual import、quality-report、write-back | 调用方预扫描携带高风险事实 |
| `TranslationRunState` 翻译运行状态 | 翻译调度器 | Agent JSON 报告、运行记录、质量错误保存 | 消费者局部异常决定全局停止语义 |
| `QualityGateResult` 质量检查结果 | Rust native quality 或统一 Python adapter | quality-report、include-write-probe、write-back gate、修复表 | Rust message 和 Python 空明细并行描述质量事实 |
| `WritePlan` 写回计划 | Rust native write plan | 文件事务执行器、写后审计、报告、诊断映射保存 | 字体、运行映射、文件替换各自定义成功/失败状态 |
| `UnitOfWork` 数据库事务边界 | persistence 事务入口 | 规则导入、写回记录、运行映射、字体记录 | repository 方法内部自提交跨表状态 |
| `AgentReportEnvelope` stdout JSON envelope | `AgentReport` 或统一报告工厂 | CLI 正常路径和错误路径 | `cli_main.print_json_error` 手写 payload |
| `NativeContract` Rust 扩展契约 | `app/native_contract.py` 和 Rust `native_contract_version()` | quality、write plan、regex、JS AST、后续 Rust 热路径 | 只检查函数存在不检查契约版本 |

目标主流程只能沿这些对象流动。任何阶段如果需要“临时兼容来源”，必须被记录为破坏性变更并在默认流程显式失败，而不是继续成功运行。

## 6. 性能重构计划

性能问题按功能缺陷处理。后续不能只说“优化了”，必须说明减少了哪些重复扫描、哪些 CPU 密集任务下沉到 Rust、哪些命令的基线发生变化。

### 6.1 必须减少重复扫描的路径

| 路径 | 当前风险 | 目标 |
| --- | --- | --- |
| `prepare-agent-workspace` / `validate-agent-workspace` | 旧目录文件和本轮 manifest 并行决定输入 | 本轮 manifest 是唯一输入清单；可选支线不做目录碰运气扫描 |
| `translate` / `run-all` / `rebuild-text-index` | 插件源码高风险扫描依赖调用方预扫描 | Workflow gate 内部拥有高风险扫描结果，同一命令复用一次扫描 |
| `quality-report(setting_overrides=...)` | 判断索引过期和重建索引使用不同配置事实 | 覆盖参数参与索引重建，TextScopeSnapshot 元信息一致 |
| `import-manual-translations` | quality_error 历史记录绕过当前 TextScope | 当前 TextScope 查询一次后决定可保存范围 |
| `quality-report --include-write-probe` | Rust gate 摘要与 Python 明细重复且漂移 | Rust 返回结构化 QualityGateResult，报告只渲染一次结果 |
| `write-back` / `rebuild-active-runtime` | 写回计划、质量门禁、字体处理、写后审计各自扫描 | Rust 生成单一 WritePlan；Python 一次应用并复用结果 |
| 规则导入 | 候选审查、译文清理、覆盖报告可能重复构建范围 | 规则导入事务中复用同一次候选范围和影响面索引 |

### 6.2 必须优先下沉 Rust 的 CPU 密集任务

| 任务 | 目标位置 | Python 保留职责 |
| --- | --- | --- |
| JS AST span/context 解析 | `rust/src/native_core/javascript_ast.rs` | 文件路径校验、插件启用状态、结果入库 |
| 占位符、源文残留、文本结构、行宽检查 | `rust/src/native_core/quality/` | 配置解析、错误文案、Agent JSON 渲染 |
| 写回协议检查和写回计划生成 | `rust/src/native_core/write_back_plan/` | 可信源路径校验、文件事务应用、失败恢复报告 |
| 大规模规则候选扫描和命中 | Rust native rules 模块 | 规则文件解析、导入事务、审查状态保存 |
| 大量文本哈希、索引 fingerprint、去重映射 | Rust 或明确的 Python 小规模胶水 | SQLite I/O、配置覆盖、索引元信息保存 |

暂时保留 Python 的 CPU 路径必须在阶段验收记录中写明：

- 输入规模边界。
- 时间复杂度。
- 为什么本阶段不下沉 Rust。
- 触发迁移的规模或性能条件。

### 6.3 性能验证方式

阶段 0 建立基线，阶段 7 写入具体耗时目标。没有基线前，不写死绝对秒数。

固定验证命令：

```powershell
uv run python scripts/benchmark_small_tasks.py
uv run python scripts/benchmark_rebuild_active_runtime.py
uv run python scripts/benchmark_active_runtime_audit.py
```

每次性能阶段交付必须记录：

- commit 或阶段编号。
- 命令和参数。
- 输入游戏夹具规模：data 文件数、插件数、插件源码大小、文本项数量。
- `ATT_MZ_RUST_THREADS` 值。
- p50 或多次运行平均值。
- 重复扫描次数或可观测计数。
- 与阶段 0 基线相比的变化。

硬性结构目标：

- 单个命令内同一 `GameData` 视图最多构建 1 次。
- 同一命令内当前 TextScope 最多构建 1 次；后续阶段只能消费快照或索引。
- 同一插件源码集合的 AST 扫描最多 1 次；结果作为 gate、索引、写回计划的共享输入。
- 写回级质量检查最多 1 次；报告和 gate 不能再重复扫描。
- 任何确需重复扫描的路径必须有代码注释、测试和阶段记录说明不可复用原因。

## 7. Python / Rust 职责边界

| 层级 | Python 职责 | Rust 职责 | 禁止状态 |
| --- | --- | --- | --- |
| CLI 和配置 | 参数解析、配置加载、环境变量校验、中文错误摘要 | 无 | Rust 读取用户配置或决定 CLI 语义 |
| 运行目录和路径安全 | 解析 `ATT_MZ_HOME`、游戏目录、可信源路径、输出路径 | 只处理已校验的相对协议或字符串 payload | Rust 直接信任未校验绝对路径 |
| 数据库事务 | 打开连接、控制事务、保存结果、schema 精确校验 | 可生成要写入的数据结构 | Rust 和 Python 同时提交同一业务事务 |
| 文本扫描和质量 | 触发扫描、传入规则和配置、渲染报告 | 执行大规模扫描、AST、质量、写回协议 | Python 复制 Rust 已实现的大规模判断 |
| 写回计划 | 校验源/目标边界、应用文件事务、回滚和报告 | 生成 WritePlan、计算写入内容、协议检查 | 字体、映射、文件写入分别定义成功 |
| LLM | prompt 组装、调用模型、重试、输出绑定 | 无 | Rust 参与 prompt 或模型输出信任判断 |
| 用户文案 | 中文摘要、下一步、Agent JSON 字段 | 返回机器可消费错误码/结构 | Rust 返回用户必须理解的长中文流程说明 |

统一规则：

- 所有 native adapter 都必须调用 `ensure_native_contract_version()`。
- native payload 语义变化必须递增 `NATIVE_CONTRACT_VERSION`。
- Python 可以做输入边界校验，但不能复制 Rust 的 CPU 密集业务判断作为第二事实来源。
- Rust 可以返回结构化明细，但不负责最终 Agent JSON envelope。

### 9.2 必须新建的业务级金丝雀

## 10. 破坏性变更清单

这些旧行为会被故意打破，因为保留它们会继续遮住当前模型。

| 旧行为 | 新行为 | 为什么值得打破 |
| --- | --- | --- |
| 缺可信源快照时首次写回自动从当前运行文件补建源快照 | 命令失败，提示重新注册或重建项目数据 | 当前运行文件不能同时是源文事实和写入目标 |
| legacy 截断候选 hash 放行规则审查 | 旧 hash 失败，必须重新审查当前完整候选 | 部分候选确认不能代表当前规则范围 |
| 旧工作区残留可选文件参与 validate | 只信任本轮 manifest | 工作区目录不是事实来源 |
| `RPG_MAKER_TOOLS_*` 作为模型服务环境变量 | 使用 `ATT_MZ_*` 当前命名；旧命名显式失败或忽略并报告 | 旧项目前缀让配置契约分裂 |
| quality_error 历史记录可直接导入手动修复 | 保存前必须属于当前 TextScope | 历史错误不是当前可写范围 |
| include-write-probe 细分字段固定为 0/空数组 | 由结构化 QualityGateResult 输出真实明细 | 报告不能同时说失败和计数为 0 |
| 写后审计失败但文件和映射已改变 | 默认回滚；若无法回滚，JSON 明确报告失败后状态 | 命令失败不能假装没有副作用 |
| 字体覆盖先于写回事务提交 | 字体操作进入 WritePlan 事务 | `restore-font` 记录必须描述成功写入 |
| 规则导入多个 repository 自提交 | 规则导入 UnitOfWork 同成同败 | 导入失败不能留下半删除译文 |
| JS AST adapter 只检查函数存在 | 检查 native contract version | 所有 Rust 热路径必须同一契约 |
| 历史修复计划继续描述当前 Agent 流程 | 当前流程只在 Skill/CLI/README 当前说明中表达 | 人类文档不能成为 Agent 契约第二来源 |
| 大文件测试锁死历史服务入口 | 按业务契约拆分并删除旧结构断言 | 测试应该保护行为，不保护坏结构 |

## 11. 第一阶段落地点

最应该先执行的是阶段 0：护栏、事实矩阵和性能基线。

原因：

1. 当前确认问题中最大的风险不是“知道要删什么”，而是删除后缺少证据判断破坏是否正确。ATT-MZ-REVIEW-020 已确认缺少完整 CLI/Agent 金丝雀。
2. 现有测试有大量 fake 和超大混合文件。直接改主流程会让测试失败难以归因，ATT-MZ-REVIEW-021 会立刻放大维护成本。
3. 用户明确关心性能慢。没有阶段 0 基线，后续无法证明重构改善的是重复扫描和 CPU 热路径，而不是主观感觉。
4. 阶段 0 不保守。它为后续删除旧 hash、旧快路径、旧补快照、旧事务边界提供可验收的破坏性护栏。

第一阶段完成后，后续 Agent 应先处理阶段 1 和阶段 2。理由是：外部报告/契约和源文/工作区事实来源是其他阶段的入口边界；如果这些边界仍分裂，后续 TextScope、写回和性能重构会继续被旧模型污染。
