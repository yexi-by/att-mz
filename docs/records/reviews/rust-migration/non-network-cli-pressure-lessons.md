# 非网络 CLI 高压收束技术复盘

## 文档定位

本文记录非网络 CLI 性能收束期间暴露出的性能和逻辑问题类型，用于后续日常开发、代码审查和回归验收参考。

本文不是当前实现百科。当前模块职责、接口边界和开发入口仍以 `docs/wiki/development/`、源码、测试和公开 CLI 契约为准。

## 高压指标暴露的问题类型

### 1. 多事实来源

部分命令曾同时依赖 Python 全量文本范围、Rust text index、数据库规则记录、workspace manifest、候选文件和命令输出摘要来判断当前文本范围或规则状态。

这种结构在小样本中容易表现正常，但在大样本、连续命令或规则变化后会出现三类问题：

- 同一命令内重复构建文本范围或重复扫描候选。
- workspace 文件和数据库当前状态不一致。
- 测试只覆盖其中一个事实来源，无法发现实际 CLI 路径漂移。

后续原则：每个命令必须明确主事实来源。能从 text index、数据库记录或 manifest 得到结论时，不应再回到完整 GameData 或 `TextScopeService.build()`。

### 2. 默认流程进入重型路径

一些轻量命令的旧实现会在默认路径中加载完整游戏数据、恢复完整译文 map、扫描插件源码 AST 或重建候选集合。

典型风险包括：

- `validate` / `import` 类命令为确认少量规则重新扫描全量候选。
- workspace 准备阶段在没有候选或已确认无需复核时仍生成临时规则文件。
- placeholder、plugin-source、MV 名字框等路径没有充分使用已有索引和门控状态。

后续原则：轻量命令只应读取完成当前判断所需的最小数据。全量扫描必须有明确触发条件，且触发条件要能被测试固定。

### 3. 空状态没有被当作有效状态

“没有候选”“没有未覆盖项”“规则已确认为空”在很多场景下是有效业务状态，不应自动等同于缺失数据或需要重新扫描。

本次暴露出的问题是，部分流程仍按旧习惯假定某些候选文件或规则文件一定存在。索引门控能证明无需处理时，正确行为应是跳过对应轮次，而不是生成空文件维持旧形状。

后续原则：空结果、已确认空规则、无候选跳过，都要作为显式状态进入 manifest、summary 或测试断言。

### 4. 测试固定旧实现形状

部分测试曾断言 prepare 阶段一定生成某类候选文件，或者某个校验命令一定经过旧扫描函数。性能收束后，这些断言不再代表用户可见契约。

后续原则：

- 测试优先固定业务结果、外部报告、状态字段和禁止回退路径。
- 不应固定非契约性的临时文件存在性、内部函数调用顺序或旧扫描中间物。
- 如果测试失败，应先判断失败点属于业务契约变化还是旧实现形状变化。

### 5. `scan_budget` 不能替代真实 CLI 证据

`scan_budget` 可以防止旧重型函数重新接入默认路径，但不能衡量以下成本：

- Python 进程启动。
- CLI 框架和配置加载。
- JSON 报告序列化和输出体积。
- 文件写入、manifest 生成和日志初始化。

本次收束中，部分小命令的业务计算已经接近目标，但墙钟仍受固定启动成本和输出成本影响。只看预算表会误判性能状态。

后续原则：性能敏感改动至少同时提供三类证据：

- 行为测试：证明结果正确。
- 静态或预算保护：证明旧重型路径没有回来。
- 真实 CLI 计时：证明用户实际等待时间下降。

## 日常开发检查项

改动非网络 CLI 本地阶段时，至少检查以下问题：

1. 该命令的主事实来源是什么。
2. 是否还会调用 Python 完整 `TextScopeService.build()`。
3. 是否会读取完整 GameData，而当前任务其实只需要索引、规则表或 manifest。
4. 是否会重复扫描 plugin-source AST、placeholder 候选、结构化 placeholder 候选或 MV 名字框候选。
5. 是否把空候选、空规则或已确认空状态误判为需要重扫。
6. 是否把大 JSON 输出、候选文件写入和 manifest 生成计入了真实 CLI 成本。
7. 测试断言的是外部契约，还是旧内部形状。

## 验收要求

涉及非网络 CLI 性能敏感路径时，建议按以下顺序验收：

1. 跑目标行为测试，确认业务输出没有变化。
2. 跑相关 `scan_budget` 或记录保护测试，确认旧重型路径没有重新接入。
3. 跑 `uv run basedpyright`。
4. 触及 Rust、schema、CLI 契约或跨模块生产代码时，跑对应 Rust 门禁和必要回归。
5. 用真实 CLI 证据记录本地阶段耗时；网络和模型等待可以剥离，但调用前后本地阶段必须计入。
6. 对低基线小命令单独说明固定成本，例如进程启动、配置加载和 JSON 输出。

## 文档和注释边界

源码注释和 wiki 只描述当前实现。批次说明、临时方案、后续接入、性能复盘和历史背景应放在计划、记录或归档中。

如果注释中出现“批次”“后续接入”“暂时保留”“先固定”等表述，应优先检查它是否已经过期。过期注释会误导后续维护者判断当前能力边界。
