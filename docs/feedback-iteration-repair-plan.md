# 试玩反馈流程修复建议

本文记录试玩反馈修复流程中暴露出的工具链问题，并给出后续迭代建议。目标不是为某个样本游戏补临时特判，而是让规则、译文保存、质量检查和写进游戏文件形成同一套可验证契约。

## 1. 设计原则

工具应遵循“外部契约为准”的原则：

- 用户、主代理、外部代理或规则文件负责声明：哪些字段可翻译、哪些协议词必须保留、哪些片段允许保留源语言。
- 工具负责校验这些声明是否结构成立：路径是否存在、规则是否命中字符串叶子、译文是否破坏控制符、写入位置是否可定位。
- 工具不得在规则已经通过校验后，再用另一套内部启发式缩小翻译范围或写入范围。
- 如果后续阶段确实不能执行前面已经通过的规则，必须 fail-fast 报错并说明差异，而不是让质量检查通过后留下源语言文本。

一句话：**工具验证契约，不发明契约；契约一旦通过校验，后续阶段必须忠实使用同一份契约。**

## 2. 本轮暴露的问题

### 2.1 文本范围不一致

同一条玩家可见文本在不同阶段出现了不同判断：

- 规则校验阶段能命中文本。
- 手动填写译文或精确重置阶段提示“不在当前可提取文本范围内”。
- 质量检查阶段显示没有问题。
- 写进游戏文件后，真实游戏文件里仍保留源语言文本。

这说明当前代码中至少存在多套“当前可处理文本范围”的计算逻辑，且它们没有共同的单一来源。

### 2.2 规则校验只证明“找得到”，没有证明“能写进去”

插件规则和事件指令规则校验现在主要证明：

- 插件名或事件指令编码匹配。
- JSONPath 能命中字符串叶子。
- 写入预演没有基础结构错误。

但它没有强制证明：

- 命中的每一条文本都会进入后续翻译集合。
- 已保存译文会覆盖对应位置。
- 写进游戏文件后真实文件会发生变化。

因此会出现“规则命中数量大于实际可写数量”的风险。

### 2.3 占位符规则容易吞掉可翻译文本

对于形如 `<Mini Label: <显示文本>>` 的插件协议，协议头需要保留，显示文本需要翻译。

如果把 `<Mini Label:` 整段开头作为必须原样保留的控制符处理，短期可以消除源语言残留检查报错，但也可能导致整条文本或部分文本从翻译集合中消失。正确做法应是结构化识别：保留协议壳，继续翻译显示文本。

### 2.4 源语言保留例外粒度过细

当前源语言保留例外主要按具体文本位置配置。对于协议词这类结构性保留内容，逐位置放行会带来三个问题：

- 同类文本越多，例外表越膨胀。
- 已经保存译文和新导出文本可能需要重复补例外。
- 规则维护成本高，容易漏掉相同结构下的其它位置。

需要支持“结构性协议词例外”，例如只在特定文本外壳中放行协议词，同时继续检查显示文本。

### 2.5 插件参数规则与实际插件配置存在覆盖差异

某些插件参数路径已经写入规则文件，规则校验也能看到，但实际写进游戏文件后仍保留源语言文本。这说明插件规则的命中结果、已保存译文记录和实际写入插件配置之间缺少覆盖矩阵。

### 2.6 插件源码硬编码文本不属于当前自动规则范围

部分插件会把玩家可见文本直接写在插件脚本中，而不是放在插件参数里。这类文本不会被插件参数规则覆盖。

当前流程需要明确区分：

- 插件参数文本：走插件规则、翻译、质量检查、写进游戏文件。
- 插件源码硬编码文本：走源码扫描、游戏侧专用处理、语法检查和真实文件反查。

如果不区分，质量检查会误以为游戏文本已经完整处理。

## 3. 已落地和仍需关注项

### P0：统一当前文本范围

当前已通过统一文本清单和覆盖审计流程对齐以下阶段的文本范围：

- 翻译任务。
- 手动填写译文导入。
- 精确重置译文。
- 源语言保留例外校验。
- 质量检查。
- 写进游戏文件。

统一结构应能解释：

- 文本在游戏里的内部位置。
- 来源类型：标准数据、插件参数、事件指令、Note 标签、术语字段等。
- 原文行。
- 是否可翻译。
- 是否可保存译文。
- 是否可写进游戏文件。
- 不能处理时的中文原因。

任何阶段都不得自己重新拼一套范围。需要使用不同范围时，必须从同一服务按明确条件筛选，并通过 `text-scope` 或 `audit-coverage` 暴露差异。

### P0：规则覆盖审计

当前 CLI 命令：

```powershell
uv run python main.py --agent-mode audit-coverage --game <游戏标题> --json
```

该命令用于比较：

- 外部规则命中的文本数量。
- 当前可保存译文的文本数量。
- 已成功保存译文的文本数量。
- 实际会写进游戏文件的文本数量。
- 命中规则但不会写进游戏文件的位置。

只要存在“规则命中但不会写进游戏文件”的玩家可见文本，命令应返回 `status=error`。

### P0：质量检查纳入规则覆盖差异

`quality-report` 不应只检查已保存译文质量，还应检查规则覆盖差异：

- 插件规则命中但没有对应可写文本。
- 事件指令规则命中但没有对应可写文本。
- 已保存译文存在，但当前写入范围不包含该位置。
- 真实游戏文件中的命中原文与项目数据库中的译文状态不一致。

这类问题应作为错误返回，禁止继续写进游戏文件。

### P1：强化规则校验输出

`validate-plugin-rules` 和 `validate-event-command-rules` 当前应暴露并维护这些字段：

- `hit_count`：规则命中字符串数量。
- `extractable_count`：会进入翻译集合的数量。
- `translated_count`：已有中文译文数量。
- `writable_count`：会写进游戏文件的数量。
- `unwritable_items`：命中但不可写的位置与原因。

只看 `hit_count` 不够。`hit_count > writable_count` 时应至少警告；如果命中的是玩家可见文本，应报错。

### P1：支持结构性源语言保留例外

当前源语言保留规则支持“在某种结构内允许保留协议词，但仍检查显示文本”。

schema 示例：

```json
{
  "structural_rules": [
    {
      "pattern": "^<Mini Label:\\s*(?P<visible>.*?)>$",
      "allowed_terms": ["Mini", "Label"],
      "check_group": "visible",
      "reason": "插件标签协议词"
    }
  ]
}
```

处理逻辑：

- `allowed_terms` 只遮蔽协议词。
- `check_group` 指定的显示文本仍需通过源语言残留检查。
- 不允许用结构规则遮蔽整句漏翻。

### P1：占位符规则校验增加“可翻译内容损失”检查

占位符规则不仅要检查往返恢复，还要检查是否误吞可翻译内容。

当前规则：

- 自定义占位符匹配片段如果包含自然语言词汇，必须提示人工确认。
- 占位符应用后，如果某条原文本来有可翻译内容，但替换后只剩占位符或标点，应返回错误。
- 对“协议壳 + 显示文本”的格式，应提示使用结构性源语言保留例外，而不是把协议壳写成会吞掉显示文本的占位符。

### P1：写入后真实文件反查

当前 CLI 命令：

```powershell
uv run python main.py --agent-mode verify-feedback-text --game <游戏标题> --input <反馈原文清单> --json
```

输入为用户反馈中出现过的源语言片段或定位表。命令应检查：

- `data` 或 `www/data` 中是否仍存在这些片段。
- `js/plugins.js` 中是否仍存在这些片段。
- 已声明需要扫描的插件脚本中是否仍存在这些片段。

如果仍存在，应返回具体文件、位置和建议处理类型：

- 外部规则缺口。
- 已保存译文缺口。
- 写入范围缺口。
- 插件源码硬编码文本。
- 图片或视频资源文字。

### P2：插件源码硬编码文本风险扫描

当前插件源码扫描命令：

```powershell
uv run python main.py --agent-mode scan-plugin-source-text --game <游戏标题> --output <工作区>/plugin-source-risk-report.json --json
```

扫描目标：

- `drawText("<文本>")`
- `drawTextEx("<文本>")`
- `TextManager` 之外的硬编码菜单文案。
- 默认输出风险报告；完整 AST 地图和候选文本只在用户确认后导出。
- 帮助窗口、确认窗口、存档窗口等常见 UI 文本。

输出只作为候选，不自动修改源码。修复方式应明确：

- 能转成插件参数规则的，优先补插件参数规则。
- 只能改源码的，生成游戏侧专用处理文件并运行脚本语法检查。
- 文档中标记这类改动可能被后续完整写入流程覆盖，直到工具支持维护它。

### P2：Skill 反馈流程增加硬性验收步骤

`skills/att-mz/SKILL.md` 和发行版 Skill 都应更新试玩反馈流程：

- 用户提供截图或原文后，先形成“反馈原文清单”。
- 补规则、补译文、质量检查和写进游戏文件完成后，必须对反馈原文清单做真实文件反查。
- 如果真实文件仍存在反馈原文，不能向用户报告本轮完成。
- 如果必须做流程外游戏侧专用处理，必须明确说明原因，并提醒后续完整写入可能覆盖它。

## 4. 推荐实现顺序

1. 抽出统一当前文本清单服务，并让手动导入、精确重置、源语言保留例外、质量检查、写进游戏文件共用。
2. 在质量检查中加入“规则命中但不会写入”的错误项。
3. 扩展规则校验输出，暴露 `extractable_count`、`writable_count` 和不可写原因。
4. 增加结构性源语言保留例外，解决协议词逐位置放行问题。
5. 使用写入后真实文件反查命令，并把它写进 Skill 反馈流程。
6. 增加插件源码硬编码文本扫描，作为插件参数规则之外的独立流程。

## 5. 测试建议

### 5.1 统一文本范围测试

构造一个中性测试游戏夹具，包含：

- 标准数据文本。
- 插件参数文本。
- 事件指令注释文本。
- Note 标签文本。

断言以下流程看到的文本位置完全一致：

- 翻译提取。
- 手动填写译文导入。
- 精确重置。
- 源语言保留例外校验。
- 写进游戏文件。

### 5.2 规则命中但不可写测试

构造一条规则能命中、但当前写入范围不会处理的文本。期望：

- `validate-*` 输出不可写原因。
- `quality-report` 返回错误。
- 写进游戏文件命令拒绝继续。

### 5.3 结构性协议词测试

使用中性协议格式：

```text
<Plugin Label: Display Text>
```

测试目标：

- 协议词可以保留。
- 显示文本必须翻译。
- 如果显示文本仍是源语言，质量检查必须报错。
- 不需要为每个具体位置写一条源语言保留例外。

### 5.4 占位符误吞测试

构造规则匹配协议壳和显示文本的情况。期望：

- 如果规则会吞掉显示文本，校验失败。
- 如果规则只保护真正的控制符，校验通过。

### 5.5 写入后真实文件反查测试

构造反馈原文清单，执行写进游戏文件后检查真实文件。期望：

- 已修复文本不再出现。
- 未进入写入范围的文本会被报告。
- 插件源码硬编码文本会被归类为源码扫描问题。

## 6. 验收标准

修复完成后，至少应满足：

- 任意规则命中的玩家可见文本，都能解释它是否会翻译、是否会保存译文、是否会写进游戏文件。
- 没有任何阶段会用私有判断推翻已经导入的外部规则。
- 质量检查通过时，规则命中的可写文本必须已经有中文译文。
- 写进游戏文件后，反馈原文清单在真实游戏文本文件中不再出现。
- 插件源码硬编码文本不会被误认为已经由插件参数规则处理。
- Skill 明确要求试玩反馈闭环以真实文件反查作为结束条件。

## 7. 当前约束

- 占位符规则只能保护协议词本体；规则吞掉显示文本时必须失败。
- 源语言残留例外同时支持位置规则和结构性规则；结构性规则只能遮蔽显示分组之外的协议词。
- 质量检查后必须通过真实文件反查结束反馈闭环，不能改回人工搜索。
- 插件源码硬编码文本只进入源码风险扫描、用户确认后的 AST 支线和游戏侧专用处理流程，不能混入插件参数规则。
- 当前规则命中但不可写时必须给出清晰错误，不能静默跳过。
