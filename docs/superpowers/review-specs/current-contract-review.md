# 当前契约 Review Spec

## 目标

审查当前源码、schema、配置、CLI、JSON 报告、错误文案、README、Skill 和测试是否只表达当前契约。

本 spec 不要求知道历史实现。审查只判断当前系统承认什么输入、什么状态、什么输出；不符合当前契约的内容应作为普通无效输入处理。

## 取证优先级

判断当前契约时按以下顺序取证：

1. 当前可执行入口、运行时代码和 `app/persistence/schema/current.sql`。
2. CLI 参数、配置模板、环境变量、JSON schema、错误码和报告字段。
3. 当前测试中固定的外部可观察行为。
4. README、wiki、Skill canonical 源和生成物。
5. `docs/history/`、发布说明、CHANGELOG 和历史设计稿只作为背景，不作为当前契约事实源。

## 审查问题

- 当前输入、状态和输出是否有清晰的唯一描述？
- 运行时代码是否为非当前形态保留专门字段、分支、loader、adapter、错误码或文案？
- 无效输入是否按当前要求失败，而不是先识别成某种历史形态再拒绝？
- 用户文案是否只说明发生了什么、影响什么、下一步做什么？
- README、Skill、配置示例和错误说明是否只描述当前流程？
- 测试是否固定当前成功行为和当前无效输入失败行为，而不是维护历史模型 helper、历史 fixture 或历史命名？
- 合法的软件生态版本、API 版本、依赖版本是否被误判为旧契约残留？

## 常见问题信号

- 当前代码、测试或文档中出现 `legacy`、`old`、`fallback`、`compat`、`deprecated`、`旧`、`回退`、`迁移` 等词，并且它们不是历史记录或发布说明。
- 错误文案解释非当前形态是什么，而不是说明当前字段要求和修正动作。
- 测试名、fixture、helper 用过去的实现作为当前断言对象。
- 当前文档引用历史设计稿作为运行契约。
- 代码为了“拒绝旧输入”新增专用识别分支。

## 输出格式

```text
范围：<文件、命令、模块或 PR>
当前契约来源：<文件:行号>
结论：PASS | FAIL | BLOCKED

发现：
- <P0/P1/P2/P3> <标题>
  证据：<文件:行号>
  问题：<为什么当前系统仍在表达非当前契约>
  建议：<删除、改成当前契约表达、补测试或无需处理>

已查无发现：
- <边界>

未验证：
- <边界和原因>
```

## 通过标准

- 当前运行时代码只接受当前契约。
- 当前错误文案只说明当前要求、当前问题和当前修正动作。
- 当前 README、Skill、配置示例和报告字段不解释非当前形态。
- 当前测试不通过历史对象证明当前行为。
- 历史说明只留在 CHANGELOG、发布说明或 `docs/history/` 等历史位置。
