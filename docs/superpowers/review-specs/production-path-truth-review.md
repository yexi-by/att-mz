# 生产链路真实性 Review Spec

## 目标

审查当前源码中的承诺是否真实落到生产链路和真实副作用。

本 spec 不依赖历史记忆。它只从当前源码、命名、报告、文档、schema 和测试中提取“当前承诺”，再从当前入口追踪真实调用链，判断承诺是否兑现。

## 当前承诺来源

以下内容都可以形成当前承诺：

- 函数、API、命令、模块或类型名，例如 `commit`、`write`、`store`、`import`、`apply`、`rebuild`、`validate`。
- CLI 帮助、README、Skill、错误文案、JSON 报告字段。
- schema、配置字段、metadata 字段、fingerprint 字段。
- 测试名、断言名、fixture 名和提交说明。

承诺不是问题。问题是承诺没有真实生产证据。

## 真实副作用

审查时优先追踪真实副作用：

- SQLite 写入、删除、替换、事务提交和回滚。
- 文件写入、备份、清理、写回和失败恢复。
- 索引、规则 store、domain state、fingerprint、metadata 和 cache 更新。
- 模型调用、prompt 组装、质量检查、报告统计和错误码输出。
- 用户可见 JSON、日志摘要和下一步提示。

## 审查步骤

1. 列出当前承诺清单。
2. 从当前 CLI、service、native adapter 或公开入口追踪调用链。
3. 标出每一步是真副作用、校验、转换、报告、no-op 还是测试专用。
4. 对比承诺 owner 和真实 owner。
5. 检查测试是否验证真实状态变化，而不是只验证 success、ok、无异常或 report 字段存在。

## 常见问题信号

- `commit_*` 没有事务写入或提交。
- `store_*` 不写 store。
- `import_*` 只校验输入，真实写入由另一个旧路径完成。
- `rebuild_*` 没有生成当前索引或当前 metadata。
- 报告字段声称清理、写入、跳过或命中，但没有真实数据来源。
- 测试只断言返回成功，不打开 SQLite、不检查文件、不验证事务失败、不查用户可见报告。

## 输出格式

```text
范围：<命令、模块、PR 或 diff>
结论：PASS | FAIL | BLOCKED

承诺矩阵：
| 承诺 | 来源 | 承诺 owner | 真实 owner | 真实副作用 | 结论 |
| --- | --- | --- | --- | --- | --- |

发现：
- <P0/P1/P2/P3> <标题>
  承诺：<当前源码中的承诺>
  证据：<文件:行号>
  问题：<承诺和真实生产链路不一致>
  建议：<接入真实链路、改名降级、删除空门面、补副作用测试>

已查无发现：
- <边界>

未验证：
- <边界和原因>
```

## 通过标准

- 每条重要承诺都能从当前入口追到真实副作用。
- API 名、报告字段、文档和测试名不夸大实现行为。
- 真实 owner 与承诺 owner 一致，或承诺被改名降级。
- 测试验证真实数据库、文件、索引、cache、报告或错误状态。
- 空门面、假成功测试和报告层拼出来的成功状态已删除或修正。
