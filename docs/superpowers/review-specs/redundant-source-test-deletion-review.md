# 冗余源码与测试删除 Review Spec

## 目标

审查当前生产契约不再需要的源码、测试、fixture、mock、stub、canary、scan budget、旧 adapter、旧 scanner、旧 fallback、旧 re-export 和只为测试存在的 helper 是否应该物理删除。

本 spec 不要求存在迁移声明。只要某段代码或测试不属于当前生产路径、当前外部契约或当前必要验证，就应判断是否删除。

## 当前生产事实源

判断是否保留时，以当前生产事实源为准：

- 当前 CLI 入口和安装入口。
- 当前 Rust/native API 中被生产路径调用的能力。
- 当前 SQLite schema。
- 当前配置模板、prompt、README、Skill 和发行包映射。
- 当前测试中必要的外部可观察行为验证。

历史 docs、records、archive、旧计划和旧设计稿可以作为线索，不能作为保留旧代码或旧测试的理由。

## 删除候选

优先审查：

- 没有生产调用方的旧实现。
- 只被测试引用的生产代码。
- 只为了旧 fixture、旧 mock、旧 stub 存在的 helper。
- 包根 re-export、兼容别名、deprecated wrapper。
- 旧 fallback、旧 scanner、旧 parser、旧 adapter、旧 validator。
- scan budget、canary 或测试账本中已不对应当前风险的条目。
- 测试名改了但断言仍证明旧机制的测试。

## 保留条件

满足以下条件之一才建议保留：

- 当前生产入口直接或间接调用。
- 当前外部契约明确暴露。
- 当前测试必须用它验证用户可见行为，且没有更贴近真实链路的替代。
- 属于内部固定逻辑，不是被替换的生产事实源。
- 属于历史记录目录，不参与当前运行契约。

## 审查步骤

1. 列出当前生产入口和关键契约。
2. 对候选文件、符号、测试和 fixture 查生产调用方。
3. 区分生产调用、测试调用、历史引用和文档引用。
4. 对只剩测试引用的对象，判断测试应改走当前真实链路还是删除。
5. 对公共导出和包根 re-export，确认是否仍是当前 API。
6. 给出删除、改写、保留或需补生产证据的结论。

## 输出格式

```text
范围：<目录、模块、测试集或 PR>
结论：PASS | FAIL | BLOCKED

删除矩阵：
| 对象 | 当前引用 | 保留理由 | 结论 | 建议 |
| --- | --- | --- | --- | --- |

发现：
- <P0/P1/P2/P3> <标题>
  对象：<文件、符号、测试或 fixture>
  证据：<文件:行号>
  问题：<为什么不属于当前生产契约>
  建议：<删除、改写测试、降级内部、补真实生产证据>

已查无发现：
- <边界>

未验证：
- <边界和原因>
```

## 通过标准

- 旧源码和旧测试被物理删除，而不是只加 guard、raise、注释或 dead code allow。
- 没有为了测试继续保留生产不用的代码。
- 保留测试只证明当前真实生产契约。
- 包根导出、public helper 和 fixture 都有当前理由。
- 删除后失败的测试只按当前契约修复，不恢复旧兼容层。
