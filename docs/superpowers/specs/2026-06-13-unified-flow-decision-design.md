# 统一流程裁决设计

## 背景

当前翻译流程已经具备很多单点检查能力，但它们分散在 `doctor`、`translation-status`、`text-scope`、`audit-coverage`、`quality-report`、`write-back` 和当前运行文件审计等命令里。Agent 需要自己拼接这些报告，再判断下一步。

这会带来几个实际问题：

- 写回级风险发现太晚。控制符、MV 虚拟名字框事实、术语写回需求和写回计划问题，可能到大规模翻译完成后才暴露。
- 命令口径分散。某些命令只检查索引可写状态，某些命令才执行完整写回级检查，Agent 容易误判。
- 翻译重试缺少项目级停机判断。连续多轮失败收益变低时，流程没有明确要求先诊断、补规则或修译文。
- 质量错误不够可行动。报告给出错误分类，但没有稳定告诉 Agent 原因、影响和下一步。
- 手动译文导入缺少保存前整包校验。错误可能在导入过程中才暴露。
- 规则或术语导入后的连锁影响不透明。文本范围、失效译文、风险确认、索引状态和写回级检查结果变化，需要 Agent 自己推理。

项目已经有公开 `doctor` 命令。因此本设计不新增第二个平行总入口，而是把现有 `doctor` 从“环境和目标游戏诊断”升级为“统一流程裁决入口”。

## 目标

1. 让 Agent 只看一个统一裁决结果，就能知道当前能不能继续。
2. 不能继续时，明确卡在规则、术语、译文、写回事实、当前运行文件还是用户授权。
3. 把完整写回级只读检查前移到正式大规模翻译和写文件之前。
4. 把低收益重试判断纳入项目命令，而不是靠 Agent 临场猜。
5. 给手动译文导入增加保存前整包校验能力。
6. 让规则和术语导入命令返回连锁影响摘要。
7. 保留现有检查命令作为细节入口，不推倒重做。

## 非目标

- 不让 `doctor` 自动修改数据库、译文或游戏文件。
- 不让 `doctor` 自动修复规则、术语或译文。
- 不删除 `translation-status`、`text-scope`、`audit-coverage`、`quality-report` 等现有命令。
- 不把人类变成运行期决策者。运行期判断服务 Agent；人类主要做源码维护、审查报告和必要授权。
- 不把本机一次性操作、私有数据或本机路径作为项目契约。

## 方案

升级 `doctor --game <游戏标题>`，让它输出统一流程裁决。无 `--game` 时仍保留环境诊断语义。

有目标游戏时，`doctor` 必须完成当前项目状态归并：

- 静态环境和配置检查。
- 目标游戏注册、源语言、当前运行目录和数据库状态检查。
- 规则审查状态检查。
- 当前文本范围和索引状态检查。
- 术语与规则、写回需求一致性检查。
- 当前可写文本的 pending、失效译文和质量错误检查。
- 完整写回级只读检查，默认执行，不写游戏文件。
- 最近翻译运行记录分析，判断是否应该继续重试。
- 写回后状态判断，提示是否需要当前运行文件审计。

`doctor` 的职责是裁决，不是替代底层报告。需要细节时，裁决结果给出下一步命令，Agent 再调用原有命令查看明细或执行修复。

## 裁决结果

`doctor` 必须给出一个主结论。主结论只允许使用以下集合：

- `blocked`：不能继续，必须先修问题。
- `ready_to_translate`：可以继续正文翻译。
- `should_stop_retrying`：继续翻译收益低，应先修规则、译文或提示词相关问题。
- `ready_for_manual_fix`：剩余问题适合导出少量修复表处理。
- `ready_to_write_back`：当前状态允许请求写进游戏文件。
- `needs_runtime_audit`：已经写过或重建过当前运行文件，需要审计运行文件。

每个裁决必须包含：

- 当前阶段。
- 是否能继续。
- 阻断原因或放行原因。
- 影响范围。
- 下一步命令。
- 是否涉及写回级风险。
- 是否需要用户授权，例如真正写文件或字体覆盖。

Agent 应该能根据主结论和下一步命令继续工作，不需要再猜多个报告之间的关系。

## 阶段判断

`doctor` 自动判断当前阶段，不要求 Agent 传入目标阶段。

阶段顺序按项目真实流程划分：

1. 准备规则和术语。
2. 小批量翻译观察。
3. 全量翻译。
4. 停止重试并诊断。
5. 手动修复。
6. 写进游戏文件前确认。
7. 写入后的当前运行文件审计。

当多个阶段同时有问题时，`doctor` 必须先报告最早阻断阶段。后续阶段的问题可以作为附加风险展示，但不得掩盖当前最应该处理的阻断点。

## 写回级只读检查

`doctor --game <游戏标题>` 默认执行完整写回级只读检查。

这项检查必须复用当前 Rust 写回级检查能力，目标是提前发现：

- 控制符被改坏。
- 文本结构无法写回。
- MV 虚拟名字框缺少当前文本事实。
- MV 虚拟名字框说话人需要术语但术语源不一致。
- 写回计划无法生成。
- 当前可写范围与已保存译文不一致。

检查不得写入游戏文件，不得替换字体，不得重建当前运行文件。性能成本必须在报告中可见，避免大游戏下静默变慢。

## 重试判断

`doctor` 必须读取最近翻译运行记录，并判断继续重试是否有意义。

基本规则：

- pending 明显下降，且质量错误没有变成同类堆积时，结论倾向 `ready_to_translate`。
- 连续多轮下降很小，且错误类型集中时，结论倾向 `should_stop_retrying`。
- 剩余数量已很小，适合人工精确处理时，结论倾向 `ready_for_manual_fix`。
- 有规则性事故、控制符风险扩大、写回级风险或术语冲突时，不允许继续机械重试。

停止重试时，`doctor` 必须给出下一步类型：

- 补普通占位符规则。
- 补结构化占位符规则。
- 修术语或名字框规则。
- 导出质量修复表。
- 导出待补译表。
- 精确重置坏译文。
- 在已授权策略内换模型。

只有涉及额外成本、接受风险、跳过问题或完整重译时，才应要求用户决策。

## 手动译文保存前校验

`import-manual-translations` 增加保存前校验模式。该模式只检查输入文件，不保存译文。

校验内容包括：

- 定位路径属于当前文本范围。
- 行数结构正确。
- 控制符和必须原样保留的片段未丢失。
- 名字框和结构化文本仍能写回。
- 导入后不会制造新的质量错误。

校验通过后，Agent 再执行正式导入。校验失败时，报告必须按条目给出失败原因、影响和建议修法。

`doctor` 不直接校验某个手动译文文件；它只在进入手动修复阶段时提示 Agent 先运行保存前校验。

## 导入后的连锁影响摘要

规则和术语导入命令成功后，不能只返回“导入成功”。它们必须返回连锁影响摘要。

摘要至少包含：

- 文本范围是否变化。
- 有多少已保存译文变成失效。
- 哪些风险确认需要重新审查。
- 是否需要重建文本范围索引。
- 是否影响写回级只读检查。
- 是否产生术语与名字框写回需求冲突。

这类摘要是导入命令自己的责任。`doctor` 负责在下一轮统一裁决中复核当前状态。

## 命令关系

`doctor` 是总入口，现有命令是细节入口。

- `translation-status` 继续负责翻译运行统计。
- `text-scope` 继续负责当前文本范围清单。
- `audit-coverage` 继续负责规则命中、文本范围、译文和写入范围一致性。
- `quality-report` 继续负责质量细节和写回级质量明细。
- `write-back` 继续是唯一正式写游戏文件入口之一。
- `audit-active-runtime` 继续负责写入后的当前运行文件审计。

文档和 Skill 中必须明确：Agent 默认先看 `doctor --game <游戏标题>` 的主裁决，再根据下一步命令进入细节报告或执行操作。

## 报告语义

`doctor` 报告必须避免状态数字吓人或口径混乱。

默认展示当前 Agent 需要处理的数量。历史运行批次统计只能作为历史字段展示，字段名和说明必须表明它不是当前阻断数量。

同一类数量必须明确范围：

- 当前可写范围。
- 当前翻译运行批次。
- 全部当前文本事实。
- 已保存但失效的译文。
- 写回级只读检查覆盖范围。

如果某项检查没有执行，报告必须明确写出未执行原因，禁止让 Agent 把“没检查”误读成“通过”。

## 测试与验证

实现需要覆盖以下行为：

- `doctor --game` 在规则缺口、术语冲突、pending、质量错误、写回级风险、低收益重试、可写回状态下给出正确主裁决。
- `doctor` 的写回级只读检查不修改游戏文件、字体或数据库译文。
- `import-manual-translations` 保存前校验模式失败时不保存任何译文。
- 规则和术语导入命令返回连锁影响摘要。
- CLI JSON 输出字段稳定，错误码和中文说明可行动。
- Skill 协议生成物无漂移。

涉及 Python/Rust 源码和可执行契约时，交付前必须按项目规则运行 `uv run basedpyright` 和全量 Python 业务测试。触及 Rust 写回级检查或 native 边界时，还必须运行 Rust 格式检查、clippy 和 Rust 测试。

## 成功标准

一次完整翻译流程中，Agent 可以在关键阶段只先运行 `doctor --game <游戏标题>`，并从报告中得到明确下一步：

- 该补规则时不会继续翻译。
- 该修译文时不会继续重试。
- 写回级风险会在正式写文件前暴露。
- 术语和名字框冲突会在规则/术语阶段暴露。
- 手动译文能先整包校验再保存。
- 规则或术语导入后，连锁影响能直接看见。

最终结果不是减少命令数量，而是减少 Agent 猜测。
