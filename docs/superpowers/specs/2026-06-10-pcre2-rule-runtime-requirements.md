# PCRE2 规则运行时重构诉求

## 背景

当前问题不是单个 MV 虚拟名字框规则的局部 bug，而是外部规则体系的结构问题。

用户和 Agent 会编写正则规则。只更换为 PCRE2 可以减少正则方言差异，但不能解决“命中之后怎么解释”的语义分散问题。真正需要收束的是：规则语法、编译校验、命中结果、分组解释、业务输出和错误报告都必须进入同一条当前主路径。

本文件记录制定正式 spec 前的诉求、边界和待设计问题。它不是实现计划。

## 总目标

完成一次破坏性规则运行时重构：

- 用户和 Agent 可写的正则规则统一使用 PCRE2。
- 正则命中后的语义解释统一进入 Rust rule runtime。
- Python 只保留命令编排、文件读写入口、配置校验、错误映射和报告组装。
- 外部规则运行事实源统一，不再让多个模块各自解释同一类规则。
- 被替换实现、被替换测试和无入口规则路径在重构完成后删除。

成功状态是：维护者只需要阅读当前规则契约、Rust rule runtime 和当前测试，就能理解外部规则如何工作；不需要知道此前项目曾经用过什么形态。

## 范围

纳入范围：

- 用户或 Agent 可写、且字段本身是正则的规则。
- 用户或 Agent 可写、且业务语义依赖正则 capture 的规则。
- 当前导入命令、数据库记录、扫描流程、报告输出和测试中与这些规则相关的路径。
- MV 虚拟名字框规则只是其中一个 domain，不是特殊例外。

不主动纳入：

- 纯内部实现细节使用的固定正则，除非它们已经形成外部规则契约。
- 与正则无关的普通枚举、路径、开关和配置字段。
- 一次性全局替换所有内部字符串处理逻辑。

## 硬约束

### 承接范围为零

当前系统只认识当前契约。输入不符合当前契约时，按普通无效输入失败。

禁止做这些事情：

- 自动转换非当前形态。
- 为非当前形态建立专门识别逻辑。
- 用专门错误说明某个非当前形态是什么。
- 在运行时保留第二条解释路径。
- 让测试继续绑定被替换实现。

### 单一事实来源

规则语义只能有一个事实源：

- PCRE2 编译、匹配和 capture 结果由 Rust 规则运行时统一负责。
- capture 到业务对象的解释由 Rust domain adapter 统一负责。
- Python 不 `re.compile` 用户或 Agent 规则。
- Python 不解释 capture 分组。
- Python 不对同一规则做第二套语义预检。

### 命令上下文决定契约

规则文件只承载当前规则内容，不承载用于切分多种形态的身份字段。

例如某个导入命令天然表示“导入当前 MV 虚拟名字框 PCRE2 规则”。文件内部不再额外声明自己属于哪套规则形态、哪种引擎或哪类运行模式。

### 无痕收束

实现完成后，当前源码、测试、schema、Skill、README、错误文案和 JSON 报告只描述当前契约。

允许出现的表达是：

- 当前字段需要什么。
- 当前输入哪里不满足要求。
- 用户下一步该如何按当前要求修正。

不允许出现的表达是：

- 系统识别了某种非当前形态。
- 系统因为某种非当前形态而失败。
- 用户需要理解此前形态才能修正当前输入。

## 架构方向

推荐方向是建立一个 Rust rule runtime，外部规则都进入同一个生命周期：

1. Python 从命令入口读取规则文件。
2. Python 做 JSON/TOML/YAML 之类的格式解析和基础字段存在性检查。
3. Python 把当前规则 DTO 交给 Rust。
4. Rust 编译 PCRE2，完成字段级规则校验。
5. Rust 把规则规范化为当前运行模型。
6. Rust 执行扫描和匹配。
7. Rust domain adapter 把 capture 解释成业务输出。
8. Python 只接收结构化结果并渲染报告。

这个 runtime 应该像一个规则内核，而不是给每个 domain 单独写一套解析器。

## Domain Adapter 原则

每个 domain adapter 只做一件事：把 PCRE2 match result 转为本 domain 的当前业务对象。

它应该明确声明：

- 输入文本来源。
- 允许的 pattern 字段。
- 必需 capture。
- 可选 capture。
- capture 到业务字段的映射。
- 失败时的当前错误码和当前用户文案。
- 输出结构。

它不应该：

- 读取 Python 的正则解释结果。
- 复用其他 domain 的隐式约定。
- 通过字符串名称猜测规则形态。
- 把测试数据形状当作运行契约。

## 数据与持久化方向

外部规则的运行事实源应收束到单一逻辑模型。

正式 spec 需要决定数据库形态，但原则已经明确：

- 不让多张规则表分别成为运行事实源。
- 不让导入文件、缓存文件和数据库同时表达可执行规则。
- 规则导入成功后，运行时只读取当前统一规则模型。
- 规则扫描报告只来自同一套 runtime 输出。

不同 domain 可以有不同 payload，但生命周期、校验、错误、启停、扫描和报告口径应统一。

## PCRE2 要求

PCRE2 更换应与规则语义收束一起做，不拆成两个互相等待的工程。

正式 spec 需要覆盖：

- Rust 侧 PCRE2 crate 选择。
- UTF-8 行为。
- 多行、大小写、dotall 等 flag 表达方式。
- capture 命名规则。
- 编译错误结构化。
- 匹配超时或 match limit。
- JIT 策略。
- 大规模扫描时的并发和内存边界。

这些要求必须进入 Rust 测试，不能只靠 Python 集成测试间接覆盖。

## Python 边界

Python 保留：

- CLI 参数和配置入口。
- 规则文件读取。
- 外部输入格式解析。
- 调用 Rust runtime。
- 把 Rust 结构化结果映射为中文报告和 JSON 报告。

Python 不保留：

- 用户或 Agent 正则编译。
- capture 分组解释。
- domain 规则语义判断。
- 与 Rust 相同目的的扫描实现。
- 与 Rust 相同目的的规则有效性检查。

## 错误处理

错误文案只说明当前事实：

- 哪个字段不满足要求。
- 当前收到的值或类型有什么问题。
- 应如何按当前契约修正。

示例口径：

- `pattern 不是有效的 PCRE2 正则：<原因>`
- `speaker capture 是必需分组，当前规则未声明`
- `body capture 命中了空文本，不能生成名字框候选`

错误文案不解释项目曾经如何处理，也不指导用户按非当前形态调整输入。

## 测试策略

Rust 测试覆盖核心语义：

- PCRE2 编译成功和失败。
- 命名 capture 提取。
- 必需 capture 缺失失败。
- domain adapter 输出结构。
- 大文本扫描边界。
- 并发扫描结果稳定。
- 资源限制生效。

Python 测试覆盖流程契约：

- CLI 参数链路。
- 规则导入成功和失败。
- JSON 报告字段。
- 用户文案。
- 数据库写入与读取结果。
- Python 不执行正则语义解释。

被替换实现绑定的测试应删除或改写为当前契约测试。测试名、fixture 名和断言文案只表达当前有效输入或当前无效输入。

## 待设计问题

正式 spec 需要继续确定：

- 统一规则模型的字段结构。
- 规则在 SQLite 中的表结构和索引。
- Rust runtime 对 Python 暴露的 API。
- 第一批纳入的 domain adapter 清单。
- 每个 domain 的 capture 命名规范。
- PCRE2 flag 配置方式。
- 规则启停、删除、重导入和报告口径。
- 大规模扫描的并发参数入口。
- 现有测试的删除、改写和新增边界。

## 验收标准

重构完成后应满足：

- 用户和 Agent 可写正则统一由 PCRE2 处理。
- Rust rule runtime 是规则语义唯一事实源。
- Python 不再编译或解释用户和 Agent 正则规则。
- 外部规则运行模型只有一套当前主路径。
- 当前文档、Skill、错误文案和测试只描述当前契约。
- 被替换实现和无入口测试已删除。
- `uv run basedpyright` 和 `uv run pytest` 通过。
- 修改 Rust 后，Rust 格式检查、clippy 和 Rust 测试通过。
